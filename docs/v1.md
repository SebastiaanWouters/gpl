# GPL v1 Implementation Plan

> Source PRD: `docs/prd-v1.md`

## Goal

Take GPL v1 base from release-candidate status to final release by shipping the full proof package promised by the PRD: a frozen and auditable spec package, a conforming Rust writer, an independent reader, a validator and strict-validator, a published conformance corpus, determinism and mmap-safety evidence, and release evidence strong enough for third-party adoption.

## Durable Decisions

These decisions apply across all phases and are not reopened by implementation work.

- **Spec authority**: `docs/spec/SPEC.md` is authoritative for all GPL v1 base on-disk behavior. If plan, PRD, and implementation differ, the spec wins.
- **Freeze policy**: GPL v1 base interoperability is frozen. Do not change required section presence, field layouts, encodings, sort rules, rejection conditions, normative semantics, or Appendix B assignments without an explicit RC reset.
- **Extension boundary**: work outside the frozen base contract stays `SKIPPABLE` and non-required. `PROVENANCE` and `BUILD_INFO` may help reproducibility and debugging but never become required for base interoperability.
- **Product definition**: GPL v1 final is a package of artifacts, not a writer-only milestone. Final release requires writer, reader, validator, corpus, determinism proof, mmap-safety proof, and release evidence.
- **Rust architecture**: crate boundaries and dependency direction are architectural constraints, not convenience choices. Shared crates may contain only frozen constants, stable types, and other truly common primitives; reader and writer logic must not drift into a shared implementation layer.
- **Writer pipeline**: the writer is built as explicit stages with invariant checks between them: source ingestion, normalized domain model, canonical GPL-ready model, and binary encoding.
- **Implementation separation**: the first reader must be meaningfully independent from the writer: separate ownership, no shared parser or serializer logic, and no shared binary-layout implementation beyond frozen constants and spec-derived tables.
- **API policy**: supported library APIs, internal crates, and CLI surfaces are explicitly identified. Internal helpers are not treated as accidental public interfaces.
- **Diagnostics policy**: public errors and validator results use stable machine-readable codes plus contextual fields such as section, offset, offending field, and relevant spec clause where practical.
- **Determinism policy**: emitted output must never depend on hash iteration order, wall clock, locale, filesystem traversal order, hostname, or other platform-specific incidental state.
- **Unsafe policy**: prefer safe Rust throughout. Any `unsafe` required for mmap or zero-copy work is isolated, documented with `SAFETY` invariants, and covered by targeted tests and review.
- **Validation model**: validator outputs distinguish container validity, section structural validity, cross-section referential validity, semantic dataset validity, and advisory-only findings.
- **Evidence model**: every base `MUST` and `MUST NOT` maps to at least one concrete proof artifact: spec clarification, test, fixture, validator check, corpus outcome, or release evidence entry.
- **Dataset policy**: Monaco and Aachen are mandatory release fixtures. Unsupported source semantics that base GPL v1 cannot represent losslessly are rejected rather than approximated.
- **Governance policy**: post-v1 compatible work is additive-only. Incompatible change requires a new major version or explicit RC reset.
- **Workflow policy**: developers and CI use the same canonical task entrypoints through a root `justfile`; release evidence is generated by automation, not by manual shell recipes.
- **Toolchain policy**: Rust toolchain, linting, formatting, and corpus/evidence schemas are pinned intentionally and upgraded deliberately.

## Planning Principles

- Prefer thin, end-to-end slices that leave behind demoable or reviewable artifacts rather than long horizontal workstreams.
- Sequence work so later implementation consumes already-reviewed contracts, corpus expectations, and evidence formats.
- Treat the conformance corpus as a product artifact shared by writer, reader, validator, CI, and release signoff.
- Keep editorial clarification and release evidence in the critical path; they are part of the product, not cleanup.
- Use review gates to catch ambiguity early instead of discovering it during late implementation.
- Make the fast developer loop explicit and cheap, and keep slow confidence loops automated in CI rather than implicit tribal knowledge.

## Phase Map

The program is organized as tracer-bullet slices that progressively turn the frozen RC contract into a releasable product package.

## Phase 0: Freeze Ledger And Release Bar

**User stories**: 1, 2, 3, 11, 12, 13, 16, 18

### What to build

Create the shared control plane for the whole program before large implementation starts. This phase turns the frozen RC contract into execution inputs: a clause-level traceability matrix, an Appendix F implementation matrix, a compatibility matrix, a deferred-extension register, fixture inventory, architecture decision records for crate boundaries and error policy, and a release evidence index skeleton mapped to `docs/spec/RC_CHECKLIST.md`.

Seed artifacts for this phase live under `docs/v1-final/` and `docs/adr/`:

- `docs/v1-final/freeze-ledger.md`
- `docs/v1-final/appendix-b-freeze-matrix.md`
- `docs/v1-final/appendix-f-implementation-matrix.md`
- `docs/v1-final/compatibility-matrix.md`
- `docs/v1-final/deferred-extension-register.md`
- `docs/v1-final/fixture-inventory.md`
- `docs/v1-final/traceability-matrix.md`
- `docs/v1-final/release-evidence-index.md`
- `docs/adr/0001-v1-architecture-and-policy.md`

### Acceptance criteria

- [ ] Every normative base requirement has an identified proof path.
- [ ] Appendix B and Appendix F freeze surfaces are enumerated and marked immutable for v1 base work.
- [ ] Out-of-scope items from the PRD and RC checklist are explicitly captured as extension-only or deferred.
- [ ] Monaco and Aachen are confirmed as mandatory release fixtures.
- [ ] The release evidence index has placeholders for every RC checklist item.
- [ ] Architecture decisions exist for crate graph, public API boundaries, diagnostics policy, determinism policy, and unsafe-code policy.

## Phase 0.5: Developer Workflow And CI Foundation

**User stories**: 7, 10, 12, 13, 14, 18

### What to build

Establish the engineering scaffolding that keeps the future Rust codebase clean while it is being built. This slice defines the pinned toolchain, root `justfile`, contributor bootstrap path, CI workflow topology, generated-artifact policy, and machine-readable formats that later phases will rely on.

Seed artifacts for this phase live at the repository root and under `docs/`, `schemas/`, `scripts/`, `manifests/`, and `.github/workflows/`.

### Acceptance criteria

- [ ] A pinned Rust toolchain policy exists, including upgrade discipline and any MSRV stance if one is chosen.
- [ ] A root `justfile` exposes canonical entrypoints for setup, doctor, fmt, lint, test-fast, test, corpus, determinism, fuzz, bench, release-check, and ci.
- [ ] CI is split into explicit stages such as PR-fast, PR-full, nightly, and release, with documented ownership and required checks.
- [ ] Generated artifacts, corpus outputs, and release evidence have a no-dirty-tree regeneration policy.
- [ ] Machine-readable schemas exist for corpus outcomes and release evidence manifests.

## Phase 1: Spec Closure And Traceability

**User stories**: 1, 2, 3, 6, 11, 13, 17, 18

### What to build

Close the remaining editorial gaps without changing frozen base behavior. The output is a spec package that an independent implementer can follow without guessing, plus a traceability matrix linking each base clause to planned writer, reader, validator, fixture, or release evidence. This phase also locks the typed model boundaries the Rust codebase will use so parsing, normalization, canonicalization, and encoding are not conflated.

### Acceptance criteria

- [ ] `docs/spec/SPEC.md`, `docs/spec/SPEC_EXPLAINED.md`, and related spec docs use consistent frozen-RC language.
- [ ] No required base field, layout, ordering rule, rejection condition, or compatibility rule is still ambiguous, draft, or implied only informally.
- [ ] Validator failure taxonomy is defined for corpus and tooling use without changing interoperability semantics.
- [ ] Every base `MUST` and `MUST NOT` is trace-linked to at least one planned artifact.
- [ ] Review confirms the spec package is sufficient for independent implementation.
- [ ] Typed boundaries are documented for source semantics, normalized domain semantics, canonical GPL-ready semantics, and encoded on-disk structures.

## Phase 2: Conformance Corpus And Outcome Contract

**User stories**: 3, 4, 5, 6, 9, 13, 14, 17, 18

### What to build

Define the corpus as the shared acceptance surface for the rest of the project. This slice includes the minimal hand-checkable valid file, Monaco and Aachen valid fixtures, invalid fixture families, expected outcome schema, failure classification rules, and corpus-runner contract that reader and validator work will target.

### Acceptance criteria

- [ ] Valid corpus includes at least one minimal hand-checkable file plus Monaco and Aachen integration fixtures.
- [ ] Invalid corpus covers header, directory, bounds, overlap, checksum, reserved-zero, dense-ID, dependency, referential, semantic, and compatibility-edge failures.
- [ ] Expected outcomes are documented as accept or reject with stable failure class where applicable.
- [ ] Unknown required features, unknown optional features, unsupported required sections, unsupported `SKIPPABLE` sections, newer `container_minor`, schema mismatches, `item_count_or_zero` conflicts, and trailing-footer-byte cases all have explicit corpus coverage.
- [ ] Writer, reader, and validator teams can consume the same corpus contract without reinterpretation.

## Phase 3: Minimal Container Slice

**User stories**: 1, 2, 3, 7, 8, 12, 13, 14

### What to build

Build the first narrow end-to-end implementation path: a Rust workspace with the intended crate graph, a minimal writer that emits a structurally valid GPL v1 file through typed model boundaries, a basic validator path that can check container framing and exact layouts, and CI gates for deterministic regeneration of the minimal fixture. This is the first proof that the frozen container contract is executable and that the codebase shape is sustainable.

### Acceptance criteria

- [ ] The Rust writer can emit the minimal valid hand-checkable `.gpl` file.
- [ ] The crate graph is established and review-approved, with clear separation between shared primitives, writer logic, validator logic, and CLI/tooling surfaces.
- [ ] Header, directory, footer, alignment, padding, and checksum behavior are verified against the spec and appendices.
- [ ] Container-level validation passes on the minimal valid file and rejects core malformed-container fixtures.
- [ ] Repeated minimal builds are byte-identical except for explicitly allowed skippable metadata, if any.
- [ ] CI runs layout checks, checksum checks, corpus format checks, and deterministic minimal-fixture generation.
- [ ] `cargo fmt`, `cargo clippy -D warnings`, docs build, and fast test entrypoints run through the same `just` commands used locally and in CI.

## Phase 4: Base Writer Vertical Slice

**User stories**: 2, 7, 9, 10, 12, 14, 15, 16

### What to build

Extend the writer from the minimal container into a full base-file producer for supported OSM and GTFS inputs. This slice covers all Appendix F required sections, canonical ordering, explicit rejection of unsupported source semantics, end-to-end builds on Monaco and Aachen, and a streaming or otherwise bounded-memory pipeline appropriate for large datasets. The purpose is to prove the frozen base contract can be emitted completely and deterministically from real inputs without collapsing the codebase into a monolith.

### Acceptance criteria

- [ ] The writer emits every Appendix F required section with the required presence, cardinality, schema, and ordering rules.
- [ ] Canonical ordering follows Appendix D, including deterministic tie-breaking behavior.
- [ ] Monaco and Aachen build successfully when their inputs are representable in base GPL v1.
- [ ] Unsupported semantics such as non-lossless transfer/pathway situations or `exact_times = 0` frequency templates are rejected explicitly rather than approximated.
- [ ] Optional `PROVENANCE` and `BUILD_INFO` handling remains `SKIPPABLE` and non-required.
- [ ] The writer pipeline preserves explicit stage boundaries with invariant checks between ingestion, normalization, canonicalization, and encoding.
- [ ] Error reporting is stable enough to expose machine-readable diagnostics with relevant context for failures.
- [ ] Memory and allocation behavior is measured on the mandatory fixtures and does not rely on unbounded accumulation by default.

## Phase 5: Independent Reader Slice

**User stories**: 1, 4, 5, 6, 9, 11, 12, 13, 16, 17

### What to build

Implement a reader that proves interoperability comes from the spec rather than from writer internals. This slice loads valid corpus files, rejects invalid ones, ignores unsupported but well-framed `SKIPPABLE` sections as required, and publishes an independence attestation explaining what was and was not shared with the writer implementation.

### Acceptance criteria

- [ ] Reader implementation satisfies the documented independence bar.
- [ ] Reader accepts valid minimal, Monaco, and Aachen corpus files.
- [ ] Reader rejects invalid corpus files for the expected reasons, including unknown required features and unsupported required sections.
- [ ] Reader ignores unsupported `SKIPPABLE` sections after framing and bounds validation.
- [ ] Independence attestation is written and review-approved.
- [ ] Dependency review confirms reader crates do not depend on writer serialization or encoding modules.

## Phase 6: Validator, Strict-Validator, And Corpus Oracle

**User stories**: 3, 4, 5, 6, 8, 13, 14, 15, 17, 18

### What to build

Turn the corpus and traceability matrix into an executable conformance oracle. This slice completes validator and strict-validator behavior across container, structural, referential, and semantic checks; stabilizes failure reporting; ties validator outcomes directly to corpus expectations and spec clauses; and exposes machine-readable output suitable for CI and release automation.

### Acceptance criteria

- [ ] Validator reports overall pass or fail for base conformance and distinguishes the planned failure classes.
- [ ] Strict-validator rejects reserved non-zero bytes, bits, enums, and flags where the spec requires rejection.
- [ ] Corpus runner compares actual outcomes to expected results for both valid and invalid fixtures.
- [ ] Every base requirement in the traceability matrix is backed by at least one validator check or another explicit proof artifact.
- [ ] Validation behavior is reproducible enough to serve as a release gate.
- [ ] Validator output includes stable exit behavior and machine-readable diagnostics for CI consumption.

## Phase 7: Determinism And mmap-Safety Hardening

**User stories**: 7, 8, 10, 13, 14, 18

### What to build

Harden the writer, reader, and validator around the production-readiness claims in the PRD. This slice proves repeated builds are byte-stable, validates offset and overlap safety under hostile inputs, demonstrates that validated files are safe for zero-copy style consumption under the spec's mmap assumptions, and verifies that any isolated `unsafe` or low-level assumptions remain justified and contained.

### Acceptance criteria

- [ ] Repeated builds for covered fixtures are byte-identical across at least three clean reruns, excluding explicitly non-deterministic skippable metadata.
- [ ] Hardening tests cover offset arithmetic, bounds, overlap, overflow, alignment, section framing, and deterministic ordering behavior.
- [ ] mmap-focused tests confirm that post-validation loading assumptions hold for valid files and fail safely for invalid files.
- [ ] CI enforces determinism checks, strict-validator runs, corpus runs, and mmap-safety regressions on the release path.
- [ ] No unresolved base-output nondeterminism remains.
- [ ] Fuzzing strategy and regression-corpus promotion rules exist for parser, reader, and validator targets.
- [ ] Any `unsafe` code paths have documented invariants, targeted tests, and explicit review signoff.

## Phase 8: Benchmarks And Real-Dataset Evidence

**User stories**: 9, 10, 12, 14, 18

### What to build

Produce the operational evidence package needed to support production-readiness claims. This slice measures build time, output size, validation time, load behavior, and representative hot-path reads on minimal, Monaco, and Aachen fixtures, plus one larger-scale rehearsal aligned with the project's stated ambitions.

### Acceptance criteria

- [ ] Benchmark methodology is documented and reproducible.
- [ ] Benchmark reports exist for Monaco and Aachen and are linked from the evidence index.
- [ ] At least one larger-scale rehearsal is completed and documented.
- [ ] Performance baselines and regression thresholds are recorded.
- [ ] Production-readiness claims in release materials are backed by measured artifacts rather than narrative assertions.
- [ ] Benchmark environments and fixture-preparation rules are controlled tightly enough that regressions are comparable over time.

---

## Phase 9: Final Evidence Index And Release Signoff

**User stories**: 4, 5, 10, 11, 13, 14, 16, 18

### What to build

Assemble the final auditable release package. This slice closes the loop from PRD to release by linking every RC checklist item to a concrete artifact, confirming that the frozen base contract was not reopened during implementation, publishing compatibility and governance notes, and making the final go or no-go decision for GPL v1 final.

### Acceptance criteria

- [ ] `docs/spec/RC_CHECKLIST.md` is fully mapped to concrete artifacts in a release evidence index.
- [ ] Final review confirms no Appendix B assignment, required section contract, or frozen base semantic rule drifted during implementation.
- [ ] Third-party implementers have published compatibility, corpus, and interoperability guidance.
- [ ] Post-v1 governance is documented as additive-only for compatible work and major-version-only for incompatible change.
- [ ] GPL v1 final is declared only if every blocker is closed or explicitly deferred outside base interoperability.

## Cross-Phase Tracks

These tracks run across phases but do not replace the phase slices.

- **Traceability**: update the clause-to-evidence matrix whenever a new artifact lands.
- **Review**: require at least one reviewer who is not the primary author for each major artifact.
- **CI**: progressively raise gates; do not wait until the end to add corpus, determinism, and hardening enforcement.
- **Governance**: re-check frozen-base boundaries whenever new ideas appear during implementation.
- **Architecture**: preserve the planned crate graph, typed boundaries, and public API limits; convenience sharing across reader and writer is not allowed to erode independence.
- **Developer experience**: keep `just` entrypoints, bootstrap docs, and generated-artifact policies accurate as the repository grows.

## Repository Guardrails

- Required status checks are defined before merge-heavy implementation starts and are tightened as phases mature.
- Generated fixtures, evidence artifacts, and machine-readable manifests must be reproducible from a clean checkout; CI fails if regeneration leaves the tree dirty unexpectedly.
- Documentation quality checks cover markdown, links, and key terminology used by the spec and release evidence.
- Toolchain, formatter, linter, and schema versions are pinned intentionally rather than floating implicitly.
- Heavy jobs are split sensibly: fast PR feedback stays quick, while fuzzing, large corpus mutation, and larger-scale rehearsal can run nightly or on release candidates.

## Developer Experience Deliverables

- A root `justfile` that is the canonical interface for local and CI workflows.
- Contributor guidance describing setup, fixture expectations, fast verification, full verification, and troubleshooting.
- A `doctor`-style workflow that verifies toolchain, fixture presence, and environment prerequisites.
- Separate fast and full verification paths so contributors know the expected pre-PR loop.
- Release automation that assembles the evidence bundle from machine-readable inputs rather than hand-maintained notes.

## Review Gates

- **After Phase 0**: confirm the plan still respects the frozen base contract and that the release bar is complete.
- **After Phase 1**: sample base clauses and verify each maps cleanly to planned proof artifacts.
- **After Phase 2**: review whether the corpus is diagnostic enough for independent teams and covers the RC checklist.
- **After Phase 4**: inspect Monaco and Aachen outputs for correctness, ordering, and explicit rejection behavior.
- **After Phase 5**: collect reader feedback on any remaining spec ambiguity; unresolved inference is a blocker or documentation fix.
- **After Phase 7**: review determinism, fuzzing, and mmap-safety evidence together before claiming production readiness.
- **After Phase 9**: compare the final evidence package directly against `docs/spec/RC_CHECKLIST.md` before release.

## Release Bars

- **Corpus**: at least one minimal hand-checkable valid file, valid Monaco, valid Aachen, and invalid fixtures for each required failure family.
- **Compatibility**: explicit fixtures for unknown optional features, unknown required features, unsupported required sections, unsupported `SKIPPABLE` sections, newer `container_minor`, schema mismatches, `item_count_or_zero` conflicts, and trailing bytes after the footer.
- **Determinism**: at least three clean byte-identical rebuilds for each covered valid fixture, excluding explicitly non-deterministic skippable metadata.
- **Hardening**: automated coverage for offset math, bounds, overlap, overflow, alignment, and framing errors, with reproducible failure capture.
- **Interoperability**: the independent reader passes the valid and invalid corpus and publishes an independence attestation.
- **Evidence**: every RC checklist item links to at least one artifact in the final evidence index.
- **Rust quality**: the crate graph is documented, public API boundaries are explicit, `cargo fmt` and `cargo clippy -D warnings` are clean, docs build passes, and diagnostics policy is implemented.
- **Workflow quality**: PR-fast, PR-full, nightly, and release workflows exist; developers and CI use the same `just` entrypoints; generated-artifact regeneration is checked automatically.
- **Safety**: any `unsafe` code is isolated, documented, reviewed, and backed by targeted tests.

## Subagent Shape

To keep execution parallel without losing ownership clarity, assign subagents by artifact family rather than by source file.

- **Spec/governance**: editorial closure, freeze verification, compatibility policy, evidence index scaffolding.
- **Corpus**: valid and invalid fixture design, expected outcomes, mutation strategy.
- **Writer**: Rust writer, deterministic emission, source rejection behavior, Monaco and Aachen builds.
- **Reader**: independent loading, compatibility behavior, skippable handling, independence attestation.
- **Validator**: validator and strict-validator behavior, corpus runner, failure taxonomy implementation.
- **Hardening/bench**: determinism checks, fuzz and mmap-safety testing, benchmark harness, larger-scale rehearsal.
- **Workflow/DX**: justfile, contributor workflow, CI pipeline design, generated-artifact policy, release automation.

## What Must Not Reopen GPL v1 Base

The following remain outside GPL v1 base finalization unless separately standardized as skippable extensions:

- pathway-aware extension layouts and semantics
- explicit frequency-template support for GTFS `exact_times = 0`
- additional non-core required sections outside Appendix F
- mutable realtime data inside the immutable base `.gpl` file

If any of these become important for a deployment, they are handled as extension work rather than as a reason to mutate frozen GPL v1 base behavior.

## Definition Of Done

This plan succeeds only when execution leaves the project with all of the following, as concrete and auditable artifacts:

- a frozen and internally consistent GPL v1 base spec package
- a conforming Rust writer for the full base contract
- an independent conforming reader
- a validator and strict-validator
- a published valid and invalid conformance corpus
- determinism and mmap-safety evidence
- benchmark and scale evidence
- a documented Rust crate architecture with stable diagnostics and explicit API boundaries
- a clean developer workflow with pinned toolchain, canonical `just` tasks, and automated CI/release gates
- an evidence index and governance package sufficient for GPL v1 final signoff

At that point GPL v1 is no longer only a strong RC. It is a formal, testable, production-ready binary format with a releasable proof package.
