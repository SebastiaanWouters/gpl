# GPL v1 Implementation Plan

## Goal

Take GPL v1 base from release-candidate status to final v1 with a production-ready implementation, a formally defined interoperability contract, and enough evidence that a third party can trust the format, build it, validate it, and read it without relying on unwritten behavior.

When this plan is executed successfully, GPL v1 must have all of the following:

- a frozen and internally consistent base specification
- a conforming Rust writer for the required GPL v1 base section set
- at least one independent conforming reader
- a validator and strict-validator with a published conformance corpus
- deterministic build evidence
- mmap-safety and robustness evidence
- benchmark and scale evidence strong enough to call GPL v1 production ready
- explicit governance for post-v1 additive work

## Planning Principles

- `docs/spec/SPEC.md` is authoritative for all on-disk GPL v1 base behavior.
- Appendix F is the canonical base section presence and dependency contract. Do not add new required base sections or change base cardinality rules without an explicit GPL v1 RC reset.
- GPL v1 base interoperability is frozen. Do not change required section presence, field layouts, encodings, sort rules, rejection conditions, or normative semantics unless GPL v1 RC is explicitly reset.
- Any work outside the frozen base contract must remain a `SKIPPABLE`, non-required extension.
- Writer, reader, and validator are separate deliverables. Final v1 is not achieved by a writer alone.
- Every base `MUST` and `MUST NOT` in the spec must map to evidence: tests, fixtures, validator checks, or interoperability proof.
- Production readiness means both correctness and operational proof: deterministic output, robust validation, realistic fixtures, and meaningful benchmark evidence.

## Starting Point

The repository is currently spec-first:

- `docs/spec/SPEC.md` defines the GPL v1 base RC contract
- `docs/spec/SPEC_EXPLAINED.md` explains the model and boundaries
- `docs/spec/RC_CHECKLIST.md` defines the current RC-to-final exit bar
- `fixtures/monaco/` and `fixtures/aachen/` provide real dataset inputs for testing and benchmarking
- the tracked repository does not yet contain the writer, reader, validator, fixture corpus, or benchmark harness needed for final v1

That means the job is not to invent the format. The job is to operationalize, prove, and ship the frozen format.

## End State Definition

GPL v1 is ready for final release only when all of the following are true:

1. The base spec is formal, internally consistent, and unambiguous.
2. The writer emits base-compliant `.gpl` files for valid source inputs and rejects unsupported semantics that cannot be represented losslessly in GPL v1 base.
3. An independent reader loads valid files and rejects invalid ones based on the spec rather than shared serializer code.
4. The validator and strict-validator enforce container, structural, referential, and semantic conformance.
5. A published corpus proves both acceptance and rejection behavior.
6. Repeated builds with the same inputs and semantics are byte-identical except for explicitly non-deterministic skippable metadata.
7. mmap-safety assumptions are exercised by tests covering bounds, overlap, alignment, and overflow behavior.
8. Release evidence shows the format is operationally useful on real datasets, not just theoretically correct.

## Execution Model

The orchestrator owns sequencing, acceptance criteria, and final signoff. Subagents do the research, implementation, review, and validation work inside each phase.

Execution should follow three standing rules:

- each phase starts with a written scope and ends with explicit exit criteria
- each major artifact is reviewed by someone other than its primary author
- no later phase can redefine frozen base semantics discovered too late; if ambiguity appears, fix the docs editorially or defer the issue outside GPL v1 base

## Workstreams

The plan runs as one program with six coordinated workstreams:

1. Spec and governance
2. Writer implementation
3. Reader and validator implementation
4. Fixture corpus and conformance evidence
5. Determinism, robustness, and mmap-safety
6. Benchmarks, release evidence, and finalization

These workstreams overlap, but the critical path is:

spec closure -> corpus design -> writer foundation -> full writer -> independent reader -> validator -> hardening -> release evidence -> v1 final signoff

## Phase 0 - Baseline Audit And Program Setup

### Objective

Turn the current RC docs into an executable program of work with no hidden assumptions.

### Step-by-step

1. Build a spec requirement ledger from `docs/spec/SPEC.md`.
2. Extract every base `MUST`, `MUST NOT`, required section rule, rejection rule, and validator obligation into a traceability matrix.
3. Build an Appendix F implementation matrix covering required sections, dependencies, cardinality, and schema versions.
4. Build a registry freeze table from Appendix B and explicitly mark base versus extension-only assignments.
5. Inventory the current fixtures in `fixtures/` and confirm whether they match the intended writer input contract.
6. Open a deferred-extension register for Section 18 items so they cannot quietly leak into the base critical path.
7. Create a compatibility matrix covering unknown required and optional features, container major and minor handling, required versus skippable sections, schema major and minor handling, `item_count_or_zero` behavior, reserved-zero enforcement, and advisory versus normative validator output.
8. Define the project release bars early: required fixtures, minimum invalid-fixture families, determinism run count, fuzz budget, CI gates, and benchmark methodology.
9. Create a risk register with at least: fixture provenance risk, deterministic tie-break risk, independent-reader scope risk, and benchmark acceptance risk.

### Deliverables

- spec traceability matrix
- base section implementation matrix
- compatibility matrix
- fixture inventory and provenance notes
- deferred extension register
- release-bar definition
- risk register

### Exit Criteria

- every normative base requirement has a planned proof mechanism
- compatibility behavior required by the spec is fully enumerated before implementation
- every out-of-scope item is explicitly marked extension-only
- open risks are visible before implementation starts

## Phase 1 - Formal Spec Closure

### Objective

Make GPL v1 base formally complete enough that implementation is an execution task, not a spec-design task.

### Step-by-step

1. Review `docs/spec/SPEC.md`, `docs/spec/README.md`, `docs/spec/SPEC_EXPLAINED.md`, and `docs/spec/RC_CHECKLIST.md` for inconsistent RC language, ambiguity, or missing cross-references.
2. Resolve editorial ambiguities in required field meaning, section presence, layout interpretation, sort order, tie-break rules, rejection conditions, and reserved-zero handling.
3. Add or tighten non-normative examples where they reduce implementation risk without changing bytes or behavior.
4. Define a stable validator failure taxonomy for corpus and tooling without changing the spec's interoperability contract.
5. Confirm that all reserved bytes, bits, enum positions, and flag positions are either assigned or reserved-zero.
6. Review Appendix B, Appendix C, Appendix D, Appendix E, and Appendix F together to ensure there is no contradiction between registries, layouts, ordering rules, and section presence.
7. Update the docs package so it consistently describes GPL v1 base as frozen, formalized, and awaiting implementation evidence rather than further format design.

### Deliverables

- revised spec docs with clarified wording only
- conformance traceability matrix linked to spec clauses
- validator failure taxonomy
- frozen registry confirmation for base interoperability

### Exit Criteria

- no required base section is ambiguous on presence, layout, ordering, semantics, or rejection behavior
- no base clause is still effectively `TBD`, draft, or pending freeze
- the spec package is strong enough for independent implementation

## Phase 2 - Conformance Corpus And Test Architecture

### Objective

Design the evidence system before the implementation grows large.

### Step-by-step

1. Define the corpus layout for valid and invalid `.gpl` files.
2. Specify one minimal hand-checkable valid GPL v1 file small enough to inspect against the appendices byte by byte.
3. Define valid integration fixtures built from Monaco and Aachen.
4. Define invalid fixture families for header, footer, directory, checksum, overlap, out-of-bounds, reserved-zero, dense-ID, cardinality, dependency, referential, and semantic failures.
5. Add specific conformance fixtures for unknown optional features, unknown required features, unsupported required sections, unsupported but well-framed `SKIPPABLE` sections, newer `container_minor`, schema mismatches, `item_count_or_zero` conflicts, and trailing bytes after the footer.
6. Define expected outcomes for each corpus entry: accept or reject, plus failure class where applicable.
7. Define how invalid fixtures are produced: handcrafted binary fixtures, controlled mutations of valid files, or both.
8. Implement the validator foundation now: stable outcome schema, corpus runner contract, and expected-result format that reader and validator work will share later.
9. Define which tests are unit, integration, property-based, fuzz, corpus-driven, or benchmark-gated.
10. Decide how corpus artifacts will be versioned, documented, and used by writer, reader, validator, and release CI.

### Deliverables

- published corpus structure
- minimal valid fixture spec
- integration fixture plan for Monaco and Aachen
- invalid mutation plan
- expected-outcome matrix
- validator foundation contract

### Exit Criteria

- every validator class has planned positive and negative coverage
- compatibility edge cases have explicit corpus coverage
- writer, reader, and validator teams are targeting the same evidence artifacts
- the corpus design is review-approved before large implementation work lands

## Phase 3 - Writer Foundation

### Objective

Build the Rust project and container machinery needed to emit structurally valid GPL files.

### Step-by-step

1. Create the Rust workspace, crate structure, test harness, and fixture plumbing.
2. Implement frozen registry constants from Appendix B in one authoritative module.
3. Implement binary encoders for header, section directory, payload framing, footer, alignment, padding, and CRC32C checksums.
4. Implement structural validation helpers used by the writer to prevent illegal offsets, lengths, overlaps, and non-zero reserved bytes.
5. Implement deterministic section ordering by `(section_type, instance)` and deterministic payload emission order.
6. Implement the manifest and string table foundation needed by later sections.
7. Add exact layout tests for Appendices A, C, and E structures used by early sections.
8. Emit the minimal valid hand-checkable fixture and validate it against the spec.
9. Wire mandatory CI gates for layout tests, corpus format validation, checksum tests, and deterministic minimal fixture generation.

### Deliverables

- Rust writer workspace
- container framing implementation
- shared registry/constants module
- exact binary layout tests
- first minimal valid `.gpl` output
- mandatory early CI gates

### Exit Criteria

- the writer emits a structurally valid minimal GPL v1 file
- header, directory, footer, checksum, and alignment behavior are test-proven
- deterministic emission exists at the container level

## Phase 4 - Full Base Writer

### Objective

Produce complete GPL v1 base files from supported source inputs.

### Step-by-step

1. Implement source normalization for OSM and GTFS inputs, including explicit rejection paths for unsupported base semantics.
2. Implement street graph extraction and required `STREET_*` sections.
3. Implement turn and complex restriction handling, including exact representation boundaries defined by GPL v1 base.
4. Implement profile sections and overlays.
5. Implement snapping sections and canonical snap-grid behavior.
6. Implement transit feed, location, service, route, pattern, trip, stop-time, service-bitset, and connection sections.
7. Implement stop/station anchors and anchor references.
8. Implement manifest population, provenance policy, and build-info policy.
9. Enforce canonical ordering from Appendix D everywhere.
10. Build Monaco and Aachen end to end.
11. Add rejection tests for pathway-aware inputs, `exact_times = 0` frequency-template semantics, or other source semantics outside frozen base support.
12. Keep `PROVENANCE` and `BUILD_INFO` optional `SKIPPABLE` sections only. They may support reproducibility and debugging, but they must never become required for base interoperability or v1 final signoff.

### Deliverables

- complete base writer
- end-to-end Monaco build
- end-to-end Aachen build
- documented rejection behavior for unsupported semantics

### Exit Criteria

- the writer emits every Appendix F required section with correct presence and cardinality
- Monaco and Aachen build successfully when inputs are representable in base GPL v1
- unsupported source semantics are rejected, never silently degraded

## Phase 5 - Independent Reader

### Objective

Prove GPL v1 interoperability is defined by the spec, not by shared writer code.

### Step-by-step

1. Lock the independence bar before implementation: separate crate, separate code owner, no shared parser or serializer logic with the writer, no shared binary-layout implementation beyond frozen constants and spec-derived tables, and a written list of allowed shared assets.
2. Implement file loading, header and footer checks, directory parsing, and section framing checks.
3. Implement loading for every base-required section.
4. Implement behavior for unknown optional features and unsupported `SKIPPABLE` sections exactly as defined by the spec.
5. Implement required rejection behavior for unknown required features, unsupported required sections, schema mismatches, structural corruption, and referential invalidity.
6. Run the reader against minimal, Monaco, and Aachen valid fixtures.
7. Run the reader against invalid fixtures and compare failures to the stable corpus outcome schema defined in Phase 2.
8. Produce a written independence attestation describing what code, tables, and logic were shared and why that still satisfies the independent-reader requirement.

### Deliverables

- independent conforming reader
- interoperability test suite against valid and invalid corpus files
- documented reader support matrix
- reader independence attestation

### Exit Criteria

- the reader loads valid files without relying on writer internals
- the reader rejects invalid files for the right reasons
- the reader demonstrates that the current docs are sufficient for independent implementation

## Phase 6 - Validator And Strict-Validator

### Objective

Implement the conformance oracle promised by the spec.

### Step-by-step

1. Implement validator modes for container validity, section structural validity, cross-section referential validity, and semantic dataset validity.
2. Implement strict-validator behavior for reserved non-zero bytes, bits, enums, and flags wherever the spec requires rejection.
3. Implement corpus runners that compare actual outcomes to expected accept/reject classifications.
4. Add controlled mutations that exercise offset overflow, overlapping sections, aliasing, invalid dense IDs, invalid spans, bad checksums, and section dependency failures.
5. Add validator checks for the compatibility matrix cases defined in Phase 0, including unknown required and optional features, required versus skippable section behavior, schema compatibility, `item_count_or_zero`, and footer completeness rules.
6. Make failure reporting stable and precise enough to diagnose whether a file is unsafe to mmap, structurally invalid, referentially invalid, semantically invalid, or advisory-only.
7. Verify that every base requirement from the traceability matrix is backed by at least one validator check or related proof artifact.

### Deliverables

- validator CLI or library
- strict-validator mode
- corpus runner
- requirement-to-check mapping

### Exit Criteria

- every major conformance rule is enforced automatically
- strict-validator behavior matches the spec
- validation results are reproducible and reviewable

## Phase 7 - Determinism, Robustness, And mmap-Safety

### Objective

Prove that GPL v1 files are reproducible and safe for zero-copy style consumption once validated.

### Step-by-step

1. Add repeated-build tests that verify byte-identical output for identical inputs and semantics.
2. Nail down every deterministic tie-break rule used by Appendix D, including provenance-derived ordering where needed.
3. Verify that string table generation, feed ordering, namespace handling, and transit ordering are stable across runs and machines.
4. Add property-based or fuzz tests for offset arithmetic, integer overflow, bounds checks, span validation, and overlap detection.
5. Add mmap-focused tests that attempt to load valid and invalid files under realistic access patterns after validation.
6. Verify that compressed versus raw section rules, logical length, stored length, and `MMAP_SAFE` behavior match the spec exactly.
7. Confirm that nondeterministic metadata, if any, remains outside base interoperability and only in skippable sections with documented behavior.
8. Enforce CI gates for determinism checks, fuzz smoke coverage, mmap-safety regressions, and corpus validation.

### Deliverables

- determinism test suite
- fuzz and property tests for structural safety
- mmap-safety regression suite
- documented reproducibility rules
- hardening CI gates

### Exit Criteria

- repeated builds are byte-identical across at least three clean reruns per covered fixture except for explicitly allowed skippable metadata
- mmap-safety assumptions are covered by automated tests
- no unresolved nondeterminism remains in base-required output

## Phase 8 - Performance, Scale, And Production Evidence

### Objective

Show that GPL v1 is not only conformant but also backed by enough operational evidence to support production-use claims.

### Step-by-step

1. Define release benchmarks for build time, output size, validation time, load time, and representative hot-path reads.
2. Benchmark the minimal, Monaco, and Aachen fixtures.
3. Add one larger-scale rehearsal aligned with the project's stated Belgium-scale ambitions.
4. Measure output size against raw inputs and track compression and structure tradeoffs.
5. Measure validation cost and mmap-backed load cost.
6. Record performance baselines and define acceptable regression thresholds.
7. Publish benchmark methodology so results are reproducible and comparable over time.

### Deliverables

- benchmark harness
- benchmark report for Monaco and Aachen
- larger-scale rehearsal report
- performance regression thresholds

### Exit Criteria

- performance claims are backed by repeatable measurements with documented methodology
- no major production-readiness risk remains unmeasured
- the project can make a supported production-readiness claim for the base format and its first implementation

## Phase 9 - Release Governance And GPL v1 Finalization

### Objective

Turn the RC checklist into a release decision backed by evidence.

### Step-by-step

1. Convert `docs/spec/RC_CHECKLIST.md` into a release signoff checklist with links to concrete evidence.
2. Assemble the final evidence package: spec docs, writer, independent reader, validator, corpus, determinism results, mmap-safety results, and benchmark reports.
3. Run a final registry freeze review to confirm no Appendix B base assignment changed during implementation.
4. Run a final base semantics review to confirm no required section behavior drifted from `docs/spec/SPEC.md`.
5. Publish corpus expectations, compatibility notes, and interoperability notes for third-party implementers.
6. Prove that unsupported but well-framed `SKIPPABLE` sections remain ignorable after framing and bounds validation.
7. Record governance for post-v1 work: additive-only minors, `SKIPPABLE` extensions for non-base work, and explicit major-version policy for incompatible change.
8. Publish an evidence index that points from each checklist item to the exact artifact that satisfies it.
9. Tag GPL v1 final only after all blockers are closed or explicitly deferred outside base interoperability.

### Deliverables

- release evidence package
- final signoff checklist
- post-v1 compatibility policy
- evidence index
- GPL v1 final release decision

### Exit Criteria

- every RC checklist item is evidenced, not merely asserted
- no unresolved ambiguity remains in any base-required field, layout, sort rule, or rejection condition
- GPL v1 can be described as formally defined and production ready with proof

## Quantified Release Bars

These bars exist to keep the plan honest. They can be raised, but they should not be lowered during execution without an explicit written decision.

- corpus minimums: one minimal hand-checkable valid file, valid Monaco, valid Aachen, and invalid fixtures for every failure family named in Phase 2
- compatibility coverage: explicit fixtures for unknown optional features, unknown required features, unsupported required sections, unsupported but validly framed `SKIPPABLE` sections, newer `container_minor`, schema mismatches, `item_count_or_zero` conflicts, and trailing bytes after the footer
- determinism bar: at least three clean repeated builds per covered valid fixture with byte-identical output, except for explicitly allowed skippable metadata
- hardening bar: property tests or fuzzing must cover offset math, overlap, bounds, and overflow behavior with a documented time budget and stable reproducer path for failures
- CI bar: the release branch must enforce corpus runs, strict-validator runs, deterministic rebuild checks, layout tests, and mmap-safety regressions
- interoperability bar: the independent reader must publish a written independence attestation and pass the valid and invalid corpus
- evidence bar: every item in `docs/spec/RC_CHECKLIST.md` must link to a concrete artifact in the final evidence index

## Review And Revision Loop

This plan should not be executed as a one-shot document. It needs explicit review and revision gates.

### Review 1 - Plan Review

Before adoption, run two reviews:

- a spec-focused review checking that the plan respects the frozen base contract
- an implementation-focused review checking that the sequence is realistic and that no enabling infrastructure is missing

Revise the plan after both reviews before beginning implementation.

### Review 2 - Contract Completeness Review

At the end of Phase 1, sample a meaningful set of base requirements and verify each one maps to a planned writer check, reader behavior, validator check, fixture, or benchmark proof. If not, revise the plan and matrix immediately.

### Review 3 - Corpus Design Review

At the end of Phase 2, verify that the corpus covers the RC checklist and that invalid fixtures are single-fault enough to diagnose. Revise fixture design before the implementation depends on it.

### Review 4 - Mid-Implementation Review

During Phase 4, review Monaco end-to-end output with a focus on correctness, canonical ordering, and source rejection behavior. Revise implementation sequencing if correctness work is getting delayed by optimization work.

### Review 5 - Interoperability Review

At the end of Phase 5, have the independent reader owner report where the spec was sufficient and where they had to infer behavior. Any inference becomes a documentation fix or a blocker.

### Review 6 - Hardening Review

At the end of Phase 7, review determinism, fuzz, and mmap-safety evidence together. If repeated builds or validation-driven loading assumptions are still weak, keep GPL v1 in RC.

### Review 7 - Release Review

At the end of Phase 9, compare the final evidence package directly against `docs/spec/RC_CHECKLIST.md`. Revise the plan one last time if any claimed readiness area still lacks proof.

## Recommended Subagent Assignments

To keep the program moving, assign subagents by role rather than by file:

- Spec subagent: traceability matrix, ambiguity review, freeze verification, governance text
- Corpus subagent: fixture design, mutation strategy, expected-outcome matrix
- Writer subagent: Rust workspace, binary layouts, data pipelines, deterministic emission
- Reader subagent: independent loader, skippable handling, interoperability tests
- Validator subagent: failure classes, strict mode, corpus runner, structural and semantic checks
- Hardening subagent: fuzzing, determinism, mmap-safety, overflow and overlap testing
- Benchmark subagent: performance harness, scale rehearsal, regression thresholds
- Release subagent: evidence package, checklist signoff, final v1 release notes

## What Must Not Block GPL v1 Final

The following remain outside GPL v1 base finalization unless separately standardized as skippable extensions:

- pathway-aware extension layouts beyond the frozen base contract
- explicit frequency-template layouts for GTFS `exact_times = 0`
- other non-core or experimental section layouts not required by Appendix F
- mutable realtime data inside the immutable base `.gpl` file

If any of these become necessary for a deployment, they must be handled as extension work without reopening GPL v1 base semantics.

## Definition Of Done

This plan is complete when executing it leaves the project with:

- a formal GPL v1 base specification package
- a production-grade writer
- an independent reader
- a validator and strict-validator
- a published valid and invalid corpus
- determinism and mmap-safety proof
- benchmark and scale evidence
- explicit release governance for future versions

At that point, GPL v1 is no longer just a promising RC. It is a formal, testable, production-ready binary format.
